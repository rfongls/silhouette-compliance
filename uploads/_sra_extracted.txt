How to Use This Template
Use Sections 1-5 to orient leadership: purpose, scope, methodology, PHI/ePHI inventory, and overall risk posture.
Use Sections 6-15 for the body of the assessment: criticality, threat model, risk matrix, safeguards review, vulnerability evidence, phishing, identity, endpoint, network, incident response, vendor risk, and site observations.
Use Sections 16-18 to convert findings into action: remediation roadmap, management response, risk acceptance, and evidence appendices.
Use the Penetration Testing Readiness Addendum only when the client has a written, authorized testing scope. A standard SRA can include vulnerability scanning and control review without being a penetration test.
Use Appendix H for internal app build-out: it maps each report section to tools, evidence, manual inputs, automation opportunities, and data model objects.

Summarize vulnerability scan, phishing simulation, identity/access, and policy/procedure review results.

2.5 Limitations and Assumptions
This report represents a point-in-time assessment based on evidence provided during the assessment period.
A vulnerability scan identifies known weaknesses but does not prove exploitability unless a separate authorized penetration test validates the attack path.
Scanner output may include false positives or items requiring business context before remediation.
Systems, vendors, applications, and PHI flows can change after the assessment; update the PHI inventory and risk register at least annually and after major changes.

Appendix G: Free Tool Options for SRA Evidence Collection (REMOVE SECTION ON FINAL)
ROW> | 
REMOVE THIS SECTION ON FINAL: This appendix is for internal planning, assessor training, and tool selection only. Do not include it in the client-facing final report unless the client specifically requests a tooling appendix. Tool names are examples, not endorsements. Verify current licensing, commercial-use restrictions, data handling, and HIPAA/BAA requirements before use. Never run scanning, phishing, wireless, OSINT, exploitation, or penetration-testing activity against any system without written authorization, approved scope, approved testing window, and emergency contacts.
Purpose. The SRA template requires evidence from interviews, policy review, PHI/ePHI inventory, vulnerability scanning, phishing simulation, identity review, endpoint review, cloud/email review, incident response, disaster recovery, business associate review, and physical site walkthroughs. The tools below can help collect or validate evidence, but they do not replace professional judgment, HIPAA analysis, client interviews, or remediation planning.
Use rules. Treat raw findings as sensitive and confidential. Confirm scope before every scan. Prefer read-only, passive, or authenticated assessment methods where available. Manually validate scanner results before assigning risk ratings. Store raw outputs in the Evidence Index or appendices, and summarize only the business-relevant findings in the report body.

Minimum Evidence to Capture When Using Any Tool
Tool name and version; date/time; operator; system used to run the tool.
Written scope and target list; approved scan window; exclusions; emergency contact.
Raw output file location; screenshots if helpful; export format; hash of evidence file if needed.
Summary of findings mapped to assets, business impact, risk rating, recommendation, and owner.
False positives, assumptions, limitations, and items not tested.
Remediation evidence and retest evidence, if retesting is authorized.
Tools That Are Helpful but Not a Substitute for a Pen Test
A vulnerability scan tells you what known weaknesses may exist. A penetration test validates whether an authorized tester can exploit weaknesses to create impact. For an SRA, these tools help gather evidence and prioritize risk; they do not prove exploitability unless the engagement has a separate, written penetration-testing scope.

H.9 Final Cleanup Instruction
Delete Appendix H before issuing a standard client-facing SRA unless the client requested an automation appendix.
Move only validated client-relevant findings into the main report sections: Risk Matrix, Technical Testing Summaries, Administrative/Physical/Technical Safeguards, Business Associate Review, and Remediation Roadmap.
Remove internal implementation notes, connector names that are not used for the client, raw target lists, credentials, tokens, tool commands, and screenshots that expose usernames or sensitive infrastructure.
Keep authorization records, raw evidence, and imports in the internal evidence repository rather than the final report body unless required as an appendix.
