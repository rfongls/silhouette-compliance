Compliance Gap Analyzer — Hosting & Integration Requirements

What it takes to move the analyzer and portal from standalone HTML prototypes to a hosted module inside your existing Next.js / Prisma / PostgreSQL application — without compromising the stateless, no-data-stored posture.
Prepared June 26, 2026 · Target stack: Node / Next.js · Prisma ORM · PostgreSQL · Stripe · Anthropic API

00Context

Three browser-based prototypes exist today. The goal is to host the compliance pieces as a feature module within your already-built premiere application, reusing its auth, database, and infrastructure rather than standing up a separate product.

AssetStateRole in hosted version
Gap Analyzer engineHaveCore logic. Calls the LLM directly from the browser today — must be repointed at a server proxy.
Portal shellHaveCustomer dashboard + admin control-board UI. Currently simulated; becomes the front-end of the module.
Server / APIBuildNext.js route handlers + Prisma. Does not exist yet — the central work.
Control-board storeBuildVersioned, admin-published control sets used for scoring. Moves from hardcoded JS into Postgres.
Guiding constraint

No uploaded documents and no patient/client data are ever persisted. The database stores accounts, billing, control boards, and (optionally) generated results — never the source files. The document is held in memory only for the duration of a single analysis request.

01Architecture at a glance

A thin, stateless request path. The browser never holds the API key and never talks to the LLM directly. Every billable action is gated server-side before any token is spent.

// Request path for one assessment Browser (portal) │ upload doc + industry (multipart, in-memory) ▼ Next.js route handler /api/assess │ 1. verify session + role (auth) │ 2. check funded / decrement (Prisma tx) │ 3. load published control board (Postgres) │ 4. call Anthropic w/ server key (ZDR tier) │ 5. persist RESULT only (never the file) ▼ Response → render report → export PDF/JSON

02The proxy — the one non-negotiable

Today the analyzer ships the Anthropic key to the browser. In production that key lives only on the server, and the route handler is what makes "pay before run" enforceable — client JS can be edited, a server check cannot.

  • Holds the key server-side via environment variables; never exposed to the client bundle.
  • Enforces the gate: no funded balance or confirmed payment → returns 402 and spends nothing.
  • Meters tokens from the provider response and writes them to the usage ledger.
  • Streams or returns the structured result; the document body is discarded when the request ends.
Provider setting, not code

Enable the Anthropic API zero-data-retention (ZDR) tier on the production account so prompts and outputs are not retained provider-side. This is what makes the "we store nothing" claim true end-to-end, not just on your servers.

03Data model (Prisma)

Reuse the existing User / Organization models from the premiere app; add the compliance-specific models below. Note the deliberate absence of any document or file model.

model ControlBoard { id String @id @default(cuid()) industry String // "health-center", "financial", ... standardKey String // "NIST", "PCIDSS", ... version Int status BoardStatus // DRAFT | PUBLISHED | ARCHIVED controls Json // the enumerated control set controlCount Int publishedAt DateTime? createdAt DateTime @default(now()) @@index([industry, standardKey, status]) } model Assessment { id String @id @default(cuid()) orgId String // the assessed org (customer's client) accountId String // billing account / team industry String status AssessStatus // PENDING|RUNNING|DELIVERED|FAILED|REFUNDED score Int? posture String? result Json? // findings, breakdown, roadmap — NOT the source doc boardCite String? // "PCI DSS 4.0.1 · rev 2026-05" refineUsed Boolean @default(false) ledgerId String? // hard link to the charge createdAt DateTime @default(now()) } model UsageLedger { id String @id @default(cuid()) accountId String kind String // "assessment" | "refine" | "purchase" | "refund" status String // "succeeded" | "failed" | "refunded" assessmentId String? // hard link to the delivered result orgsBilled Int? amountCents Int? inputTokens Int? outputTokens Int? stripeRef String? createdAt DateTime @default(now()) }

Declare the datasource as PostgreSQL and add the two enums the models reference. Json compiles to Postgres jsonb, DateTime to timestamp, and each enum to a native Postgres ENUM type.

datasource db { provider = "postgresql" url = env("DATABASE_URL") } enum BoardStatus { DRAFT PUBLISHED ARCHIVED } enum AssessStatus { PENDING RUNNING DELIVERED FAILED REFUNDED }

Auth models (Auth.js + Prisma adapter)

Google and Microsoft sign-in is OIDC, not SAML — the light kind of SSO. Auth.js (NextAuth) ships Google and Microsoft Entra providers plus a Prisma adapter that writes identity straight into the same Postgres. Add its standard models; map your customer / admin role and account/team onto User.

model User { id String @id @default(cuid()) email String @unique name String? image String? role String @default("customer") // customer | admin accountId String? // billing account / team accounts Account[] sessions Session[] } model Account { // OAuth link (Google / Microsoft) — Auth.js id String @id @default(cuid()) userId String provider String // "google" | "microsoft-entra-id" providerAccountId String access_token String? user User @relation(fields: [userId], references: [id], onDelete: Cascade) @@unique([provider, providerAccountId]) } model Session { id String @id @default(cuid()) sessionToken String @unique userId String expires DateTime user User @relation(fields: [userId], references: [id], onDelete: Cascade) }
Integrating into the primary app DB

These models live in the same PostgreSQL instance as the premiere app. If a User already exists there, extend it with role / accountId instead of duplicating, and add only the compliance models (ControlBoard, Assessment, UsageLedger) plus Auth.js's Account / Session. Run as a single additive Prisma migration.

05Auth, roles & tenancy

Inherit from the premiere app. Two roles drive the UI and the route guards.

  • Customer — queue, pay, run, view history, one refine per report, export. Scoped to their account/team.
  • Admin — control-board fetch / publish / versioning, snapshot export, per-org rate, accounts & usage. Every admin route re-checks the role server-side.
  • Tenancy — bill and scope by account/team (a health system or association), not individual users, so a team shares balance and report history. SSO (Google / Microsoft) for enterprise buyers.

06Payments & gating

Flat, per-organization pricing makes "fund first" clean: collect the full amount up front, then run.

  • Unit: one assessment = one organization (any number of its documents), one report, all exports, one refine.
  • Two flows: pay-at-run (one-time Stripe PaymentIntent) and prepaid packs (3 / 10) that decrement a ledger balance.
  • Trust the webhook, never the browser, as proof of payment. The run unlocks on payment_intent.succeeded.
  • Cost guard: cap document size and standard count per run so worst-case token spend is bounded and the flat price stays high-margin.
  • Auto-refund the unit on a failed run so a broken job is never charged.

07Billing history, receipts & dispute protection

The defense against "I was billed but nothing ran" or "I couldn't save it." Every charge is structurally tied to a delivered or refunded result, and every result stays re-downloadable.

Two records, always linked

  • Billing ledger — append-only, immutable: timestamp, account, amount, org count, Stripe reference, token usage, and status. This is the audit evidence.
  • Report history — customer-facing list: org, date, score, board version, status, and a re-download link for the report / deck / JSON.
  • Hard link — each UsageLedger charge references its Assessment and vice versa, so a charge can never exist without a delivered or refunded result attached.

Persist the result, not the document

Store the generated output (findings, score, roadmap) so it is always retrievable from History — surviving a closed tab, a blocked download, or a new device. Auto-trigger the download on completion as a convenience, but the durable retained copy is the real protection. The source document is still never stored.

Status lifecycle closes the gap

An assessment moves pending → running → delivered, or failed → refunded. Payment is captured only after the run unlocks (webhook-confirmed), and a failed run auto-refunds and shows as refunded in History. There is no state in which a charge sits next to a missing report.

Privacy statement, stated precisely

"We never store your uploaded documents. Generated reports are retained in your account so you can re-download them, and you can delete them at any time." This keeps the strong posture while making the history feature honest.

08Control boards — store, version, refresh

The single biggest cost lever. Pre-fetch and cache control sets once, shared across all customers, instead of paying to enumerate them per run.

  • Reference data, not customer data — public framework controls (PCI, NIST, …). Storing them does not touch the no-data-stored posture.
  • Admin workflow: Fetch → DRAFT → review → Publish as version N. Customers always score against the latest PUBLISHED version.
  • Versioned + date-stamped so a report can cite "PCI DSS 4.0.1, control set rev. 2026-05" and a past assessment is reproducible.
  • Scheduled snapshot export (weekly / monthly cron) writes an archived copy — your durable record and the admin-export feature already designed in the portal.

09Local testing (before hosting)

The entire flow — account creation, SSO, purchase, run, history — runs on localhost. Nothing needs to be deployed to validate the model.

Google / MS SSOBoth allow localhost redirect URIs. Register an OAuth app in Google Cloud Console and Azure/Entra; paste client ID + secret into Auth.js. Full sign-in works locally.

10Productionizing the prototypes

  • Analyzer: replace direct Anthropic calls with fetch('/api/assess'); remove the client-side key entirely.
  • Portal: swap simulated data for real route calls; move localStorage state into Postgres-backed queries; keep the existing UI/brand.
  • Exports: the HTML report / deck / JSON generators port as-is; optionally move PDF rendering server-side for consistency.
  • Industries & standards: the existing config maps cleanly onto ControlBoard.industry / standardKey.

11Phased build

Phase 1 — Accounts + take money safelyMVP

Auth.js with Google + Microsoft (OIDC) sign-in, the proxy route, Stripe one-time charge, and webhook gating. Light SSO is in from the start to cut signup friction; SAML/SCIM stay demand-triggered in Phase 4.

Phase 2 — Consistent scoringCore

ControlBoard model + admin fetch/publish/versioning + scheduled snapshot export. Removes per-run fetch cost.

Phase 3 — Repeat-buyer convenienceGrowth

Assessment + billing history with re-downloadable results, prepaid packs, immutable usage ledger, status lifecycle with auto-refund, dashboard wired to real data.

Phase 4 — Enterprise readinessScale

Team accounts, SAML SSO + SCIM (demand-triggered), sales-assisted invoicing for gov / finance buyers. The operator-run workflow continues in parallel throughout.

Bottom line

The true MVP is the proxy + auth + Stripe trio — everything else is enhancement layered on the same stateless path. Because the premiere app already provides Postgres, Prisma, and auth, the compliance module is largely new route handlers, three Prisma models, and repointing two existing front-ends.
